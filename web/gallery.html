Section 2 · Three cities, one band

Now the smallest interesting case: one scalar key, copdem30m.elevation_mean, painted across three coastal cities at 1 km resolution. The point isn't pretty cartography. The point is that you can paste any of these URLs into a colleague's chat and they'll see exactly the same bytes.

What the colour means. A five-stop terrain ramp from cream (low elevation) to deep forest (high). The legend in each figure carries the actual range in metres so the same hue means different absolute heights in different places. Why squares. Each tile is one of an 8×8 grid of cell64 samples covering the city's bounding box; the responder geocodes the place, walks the grid, recalls each cell, and paints. Latency. Cold ~3 s (first agent pays the materialiser cost); warm ~150 ms.

Section 3 · Hunter mode

Most agent questions name a place. A few don't. "Find me where the algal bloom is." "Where is the deforestation happening?" These don't anchor on a cell — they sweep a region looking for an event. emem now answers them directly.

The path is one POST. The responder picks the right detection algorithm from /v1/algorithms, samples up to thirty-two cells from the named region, recalls the algorithm's primary scalar input at each, and returns the top eight ranked cells. Each carries the recalled value, a fact CID, and a Sentinel-2 scene URL. Same receipt machinery as any other primitive.

Four short case studies follow. Every cell, every value, every CID below was returned by a live /v1/hunt call. Click any scene tile to open the Sentinel-2 RGB for that cell.

Case 1 · Algal bloom · Lake Erie

Chlorophyll-bright water along the south shore

A summer satellite pass picks out the same near-shore band any Erie boater would name. The eight ranked cells sit between Cleveland and Buffalo, NDVI climbing past 0.7 over water — the chlorophyll-NDCI proxy's bloom threshold.

The detection algorithm is algal_bloom_chlorophyll_ndci@1. It gates on NDWI > 0 (water) and turns the residual vegetation index into a chlorophyll-a estimate. The full formula is in the algorithm card; the ranking below uses NDVI alone as a cheap primary scalar, and the hotspot positions match the NOAA GLERL bloom bulletin's annual footprint.

Case 2 · Deforestation · Borneo

Negative NDVI on what used to be forest

The strongest hotspots over Borneo come back with NDVI below zero — bare ground or active clearing where the canopy was. The eight cells trace the well-known palm-oil concession edges in Sabah and West Kalimantan.

The algorithm is deforestation_alert_ndvi_drop@1: a 60-day NDVI drop on cells the Hansen 2000 tree-cover layer declares forested. The ranking below sorts cells by lowest NDVI so the headline is the most-cleared cell first. The full algorithm gates on the Hansen baseline so primary-forest cells are kept and pre-existing bare ground is filtered out — apply the formula via the algorithm card to score every cell properly.

Case 3 · Sediment plume · Sundarbans

Red-band reflectance over the Ganges delta

The Bay of Bengal's brown water has a specific signature: bright red, dim near-infrared. The hotspots cluster inside the Sundarbans mangrove channels where the river silt hasn't dropped yet — the canonical place to look for muddy water on a planet-scale.

The algorithm is water_turbidity_red_band@1. The primary scalar is Sentinel-2's red band over a water-masked cell. The values below sit near and above 1.0 — saturated red reflectance, which is the cheapest available turbidity proxy short of swapping in a hyperspectral instrument.

Case 4 · Inundation extent · Karachi coast

Sentinel-1 backscatter below the open-water threshold

The C-band radar threshold for open water — about -16 dB VV — turns the Korangi Creek mouth into a tightly clustered grid of low-backscatter cells. Eight hotspots, every value in single-digit negative dB, every cell within 200 m of saltwater.

The algorithm is flood_extent_sar_threshold@1. It classifies cells with VV below the operational threshold as water — useful post-event for inundation extent (cyclone Maha 2023, Sindh monsoon flood 2022) when a return scene is needed under cloud cover. The example below sweeps Karachi's bbox; on a dry day the same call surfaces the city's permanent water (creeks, port basins) as a sanity-check on the algorithm.

Section 4 · The honest limit

The previous four cases worked. This one doesn't, and the gallery is the right place to be specific about that.

Case 5 · Oil spill · Persian Gulf

What the responder says when it can't answer

An oil-slick detector is not currently in the algorithm registry. The hunter call below returns a structured not_yet_implemented with pointers at the closest available physics. No hallucinated hotspots, no fabricated receipts — the agent sees the limitation as data, not a 500.

$ curl -sS -X POST https://emem.dev/v1/hunt \
    -H 'content-type: application/json' \
    -d '{"event":"oil_slick","region":"Persian Gulf"}'

{
  "status":               "not_yet_implemented",
  "event":                "oil_slick",
  "region_anchor":        "Persian Gulf",
  "message":              "oil-slick detection has no registered
                          algorithm at this responder.",
  "closest_algorithms": [
    {
      "key":         "flood_extent_sar_threshold@1",
      "rationale":   "Slicks dampen capillary waves;
                     Sentinel-1 VV reads them as low
                     backscatter."
    },
    {
      "key":         "water_turbidity_red_band@1",
      "rationale":   "Heavy slick films carry emulsified
                     sediment the red band sees."
    }
  ]
}

The fix is two pieces of work, both on the public roadmap. The first is a Sentinel-1 surface-roughness anomaly algorithm — the same physics the closest-algorithms pointer describes, but calibrated for slick detection (Vincent-Liu et al, IEEE J-STARS 2020). The second is the supporting reference dataset: a few hundred confirmed slick cells from ITOPF case histories and the ESA Sea-Track Wind product to train the anomaly threshold against.

Why this section exists. A demo that smiles through everything teaches an agent to smile through everything. By making the limitation citable — same schema as a real answer, with two real algorithms the agent can fall back on — we keep the protocol's promise: when emem doesn't know, it says so in a shape that survives onward composition.

Section 5 · A worked answer

The point of signing isn't the math. The point is what you can paste into a chat and have someone else replay.

Question → bytes → receipt

Five lines. One agent asks; another agent verifies.

$ curl -sS https://emem.dev/v1/elevation?lat=19.0760&lng=72.8777

{
  "value":         14.0,
  "unit":          "m_msl",
  "source":        "copdem30m",
  "fact_cid":      "bafy2bzaceaqxfn4obg3orodvm6ws3lnxv5fbzqbtku2c3ye2pjp7zkx2cmpbg",
  "receipt": {
    "responder":      "777er3yihgifqmv5hmc2wwmyszgddzderzhsx6rex4yoakwomvka",
    "signature":      "…64 bytes ed25519…",
    "served_at":      "2026-05-14T05:51:12Z",
    "merkle_proof":   "…6 sibling hashes…"
  }
}

The agent that received this response now has three things: a number (14 m above mean sea level at South Mumbai), a 64-byte proof, and a 26-character handle. The handle is the fact CID. Anyone with the handle can re-run the math:

$ curl https://emem.dev/v1/facts/bafy2bzaceaqxfn4obg3orodvm6ws3lnxv5fbzqbtku2c3ye2pjp7zkx2cmpbg

# or, with no callback at all:
$ node verify_offline.js receipt.json pubkey.b32
verified: signature checks against responder 777er3yihg…

The signing key is /.well-known/emem.json. The verifier lives at /verify — open it, paste a fact CID, watch @noble/curves recompute the signature inside the browser tab. No callback to the issuer.

Why this matters. Two LLM agents disagreeing about whether South Mumbai is 14 m or 4 m can point at the CID and stop arguing. Two insurance underwriters disagreeing about whether a parcel is in the flood plain can do the same. The CID is the citation.
