Section 3 · Hunter mode

Some questions ask about an event rather than a place: "find algal blooms in Lake Erie", "where is deforestation happening in Borneo". These don't anchor on a single cell. They sweep a region looking for a pattern.

The POST /v1/hunt endpoint takes an event keyword and a region (free-text name or polygon_bbox), picks the matching detection algorithm from /v1/algorithms, samples up to thirty-two cells inside the region, recalls each cell's primary scalar band (and a gate band where the algorithm needs one), and returns the top eight cells ranked by that scalar. Each hotspot row includes the cell64, the value, a fact CID, and a Sentinel-2 scene URL.

Four case studies follow. The cells, values, and CIDs below were returned by live /v1/hunt calls. Click a scene tile to open the Sentinel-2 RGB for that cell.

Case 1 · Algal bloom · Lake Erie

Water-gated NDVI returns near zero in May

Sampled in May 2026, before Lake Erie's summer cyanobacteria season. The detector ranks chlorophyll-NDCI over water-masked cells; in May the values sit near zero because there is no active bloom.

The algorithm is algal_bloom_chlorophyll_ndci@1. It requires NDWI > 0 to confirm water before scoring NDVI as a chlorophyll proxy. The gate is applied to the candidate cells before sorting, so every hotspot below sits over water. The NDVI values are near zero because Lake Erie's bloom intensity peaks in July–September per NOAA GLERL; the same call in August will return positive NDVI over the same cells.

Case 2 · Deforestation · Borneo

Negative NDVI on cells the Hansen baseline says were forested

The top hotspots over Borneo come back at NDVI below zero: bare ground or active clearing on the cells that Hansen 2000 tree-cover marks as forested. They fall along the palm-oil concession edges in Sabah and West Kalimantan.

The algorithm is deforestation_alert_ndvi_drop@1: a 60-day NDVI drop on cells whose Hansen 2000 tree-cover layer reads forested. The ranking here sorts by lowest NDVI so the most-cleared cell ranks first. The full algorithm additionally requires the Hansen baseline to exceed a threshold (so pre-existing bare ground is excluded); apply the formula from the algorithm card to score every cell with the gate intact.

Case 3 · Sediment plume · Sundarbans

Red-band reflectance over the Ganges delta

Brown river water carries elevated reflectance in Sentinel-2's red band and lower reflectance in the near-infrared. The hotspots fall inside the Sundarbans mangrove channels where the suspended silt has not yet settled.

The algorithm is water_turbidity_red_band@1. The primary scalar is Sentinel-2's red band (B04) over a water-masked cell. The values below approach 1.0, which is at the saturation edge of the red band; without a hyperspectral follow-up, this is the cheapest available turbidity proxy.

Case 4 · Inundation extent · Karachi coast

Sentinel-1 backscatter below the open-water threshold

C-band VV below roughly -16 dB is the operational open-water threshold over land. The Karachi sweep returns eight hotspots in single-digit negative dB; all sit at the Korangi Creek mouth, within 200 m of saltwater.

The algorithm is flood_extent_sar_threshold@1. Cells with VV below the operational threshold are classed as water. Used post-event for inundation extent (cyclone Maha 2023, Sindh monsoon 2022) where a Sentinel-1 pass cuts through cloud cover that Sentinel-2 cannot. On a dry day the same call returns the city's permanent water surfaces (creeks, port basins) — a useful sanity check.

Case 5 · Oil spill · Persian Gulf

What a 200 response looks like when the algorithm is missing

There is no oil-slick detector in the algorithm registry today. The hunt endpoint returns status: not_yet_implemented, lists the closest available algorithms, and explains why each is close. No hotspots are returned. No receipt is forged.

$ curl -sS -X POST https://emem.dev/v1/hunt \
    -H 'content-type: application/json' \
    -d '{"event":"oil_slick","region":"Persian Gulf"}'

{
  "status":               "not_yet_implemented",
  "event":                "oil_slick",
  "region_anchor":        "Persian Gulf",
  "message":              "oil-slick detection has no registered
                          algorithm at this responder.",
  "closest_algorithms": [
    {
      "key":         "flood_extent_sar_threshold@1",
      "rationale":   "Slicks dampen capillary waves;
                     Sentinel-1 VV reads them as low
                     backscatter."
    },
    {
      "key":         "water_turbidity_red_band@1",
      "rationale":   "Heavy slick films carry emulsified
                     sediment the red band sees."
    }
  ]
}

The fix is two pieces of work, both on the public roadmap. The first is a Sentinel-1 surface-roughness anomaly algorithm — the same physics the closest-algorithms pointer describes, but calibrated for slick detection. The second is the supporting reference dataset: confirmed slick cells from ITOPF's public archive and Copernicus CleanSeaNet alerts to anchor the anomaly threshold against ground truth.

The response shape is the same one a successful hunt would use; only the status field changes. An agent that calls /v1/hunt with an unsupported event therefore sees a typed result, not an HTTP error or an empty array. The two pointer algorithms in closest_algorithms[] are real registry entries; an agent can call either one and get a partial answer for the underlying physics.

Section 7 · EUDR Due Diligence Statement

A polygon in, an Annex II envelope out. The forest definition follows Regulation (EU) 2023/1115 Article 2(4) as written — ten percent canopy, not the FAO thirty.

Case · Coffee plot, Amazonas, Brazil

Signed compliance verdict from forest baseline plus loss-year

The endpoint runs eudr_compliance@1 over every cell inside the plot polygon. Forest baseline is the JRC GFC2020 V3 single-COG mirror (the EU Commission's expected non-binding baseline) plus Hansen treecover2000 at the regulation's ten- percent threshold. Sims et al. 2025 driver attribution and RADD Sentinel-1 alerts layer on as refinement when the connectors are wired; today they sign Absence with a structured NotImplemented reason and the verdict comes from JRC + Hansen alone.

The plot here resolves to a 100 km2 bbox over primary Amazon forest. The Article 2(28) dispatch picks POLYGON (the plot is well above the four-hectare threshold; coffee is not cattle). The endpoint samples sixteen cells inside the polygon, recalls JRC forest_2020 plus Hansen treecover2000 and lossyear at each, evaluates the eudr_compliance@1 logic, and aggregates with a 0.1 % de-minimis threshold so a single bad pixel on the polygon edge does not flip the plot to FAIL.

What an auditor sees in the response. The fact CIDs at every cell are signed by the responder under its Ed25519 identity. Quote any one of them at /verify and the browser recomputes blake3 + ed25519 in JavaScript — no server call needed to confirm the signature held.

What the engine does not claim. The legality_module field defaults to "none" and the response carries the explicit disclaimer that Article 9(1)(b) legality verification — land tenure, free prior and informed consent, country-of-origin law compliance under Article 2(40) — is structurally out of Earth-observation scope. The geolocation block at the cell level is signed; the legal layer above it is not. Operators submitting to TRACES NT pair this DDS with a partner module that handles those questions.

An earlier vintage of eudr_compliance@1 (removed 2026-05-17) used the FAO 2020 Global Forest Resources Assessment thirty-percent canopy threshold. The regulation as written uses ten. Plots between ten and thirty percent canopy — frequently smallholder agroforestry under cocoa or coffee — fall inside the EUDR scope and would have been silently dropped under the older threshold. The current algorithm tests both the JRC GFC2020 V3 baseline (the Commission's expected non-binding map) and Hansen treecover2000 at ten percent.
