# emem.dev/humans — knowledge constellation for AI agents

> A single-page exploration of the emem corpus. Every visible thing carries
> structured data attributes (cell64, band, fact_cid, tslot) so a scraping
> LLM can extract everything from the rendered DOM. Every /v1/* call the
> page makes is printed in a live console pane — read the page, learn the
> API by observation. Sibling JSON twin: /humans.json.

## Endpoints discoverable on this page

- `GET /v1/discover` — typed bootstrap, manifest CIDs, primitive list
- `GET /.well-known/emem.json` — responder pubkey, key epoch, manifests
- `GET /v1/coverage` — `cells[].{cell64, fact_count, lat_deg, lng_deg}` (densest first)
- `GET /v1/coverage_matrix?limit=200` — per-band facts_count, last_attested_unix_s, family, tempo_seconds, history bounds
- `GET /v1/data_availability` — per-band tempo_seconds + history_available_{from,to}_unix
- `GET /v1/bands` — band registry (35 cube slots, 118 materializer-wired band names)
- `GET /v1/algorithms` — algorithm registry (155 composition recipes; each carries `parameters` + `learned_from` provenance)
- `GET /v1/algorithms/{key}` — per-key drill-down with formula, inputs, citation, accuracy_band
- `GET /v1/sources` — fetch-source registry (43 declared schemes; 12 wired data connectors)
- `GET /v1/manifests` — the manifest CIDs (registry, bands, algorithms, sources, schema, topics, lcv-1, alphabet)
- `GET /v1/schema` — canonical CBOR wire schema
- `GET /v1/topics` — 26 declared topic clusters with `live_bands_by_topic` filtered to this responder's wired materializers
- `GET /v1/agent_card` — agent-readable summary, live counts, manifest CIDs
- `GET /v1/tools` — 55 MCP tools with `when_to_use` strings + four behavioural annotations
- `GET /v1/quickstart` — copy-paste snippets for the canonical flows
- `GET /v1/cells/{cell64}/info` — bbox, centre, approximate metres
- `GET /v1/cells/{cell64}/scene.png` — optional Sentinel-2 thumbnail
- `POST /v1/locate` — `{q}` → cell64 + bbox + topic-grouped band index at that cell
- `POST /v1/ask` — `{q}` → topic-routed answer + signed facts; foundation-embedding fan-out fires on Similarity / Change intents and adds a `foundation_embeddings` envelope with per-encoder neighbours + cross-encoder consensus voting
- `POST /v1/recall` — `{cell, bands?}` → signed `facts[]` + receipt; auto-materializes on miss
- `POST /v1/recall_many` — `{cells[≤256], bands?}` → per-cell facts + batch receipt
- `POST /v1/recall_polygon` — `{polygon|place, bands, include?}` → fanout fact set + receipt; `include: ["ftw_fields"]` attaches FTW per-field polygons
- `POST /v1/find_similar` — `{key|cell, band?, k, mode?}` → k-NN; modes `cosine` | `hamming` | `hamming_then_rerank` (EWMA-adaptive triage). Auto-derives `geotessera.bin128` from the cosine sibling on miss
- `POST /v1/trajectory` — `{cell, band, window}` → ordered tslots → values + receipt
- `POST /v1/compare` — `{a, b, bands?}` → per-band distances + receipt
- `POST /v1/compare_bands` — `{cell, a, b, tslot_a?, tslot_b?}` → cross-band cosine / delta + signed verdict
- `POST /v1/diff` — `{cell, band, tslot_a, tslot_b}` → signed DerivativeFact
- `POST /v1/field_boundaries` — `{bbox|place|cell}` → Fields of The World per-field polygons (CC-BY-4.0)
- `POST /v1/query_region` — `{geometry|bbox}` → region aggregate; default `max_cells` is bbox-area-derived (clamped [64, 1024])
- `POST /v1/backfill` — `{cell, band, start_unix, end_unix, max_facts?}` → historical materialization
- `POST /v1/verify` — `{cell, claim}` → signed verdict + evidence fact CIDs
- `POST /v1/verify_receipt` — `{receipt}` → `{valid, preimage_blake3_hex, signer_pubkey_b32}`
- `POST /v1/temporal_route` — `{cell, intent?, bands?, limit?}` → recipe plan (which bands at which lookback windows)
- `POST /v1/heat_solve` — 2-D explicit FTCS heat over a 3×3 MODIS LST stencil; CFL-stable
- `POST /v1/wave_solve` — 1-D shallow-water along the seaward bathymetry gradient; 422 on land-locked
- `POST /v1/jepa_predict` — closed-form NDVI AR(2) seasonal predictor (α=0.6, β=0.3, γ=0.1, not learned)
- `POST /v1/jepa_predict_v2` — Tessera embedding dynamics; currently short-circuits to last attested vintage and signs receipt with `untrained_baseline`
- Domain shortcuts (locate → recall → polygon-aggregate by place name):
  - `POST /v1/at` (multi-band), `/v1/ndvi`, `/v1/air`, `/v1/lst`, `/v1/soil`, `/v1/water`, `/v1/forest`, `/v1/weather`, `/v1/elevation`
- `GET /verify` and `GET /verify/{fact_cid}` — in-browser ed25519 receipt verifier; reconstructs the canonical preimage and runs the signature math with @noble/curves + @noble/hashes (no callback to issuer)

## Data attributes (DOM scraping)

Every star, fact card, receipt row, and registry node carries:

- `data-emem-cell="defi.<bigram>.<bigram>.<bigram>"` — cell64 string
- `data-emem-band="<band-key>"` — band key (e.g. `copdem30m.elevation_mean`)
- `data-emem-fact-cid="<base32-32B>"` — content-addressed CID
- `data-emem-tslot="<unix_s>"` — Unix-epoch tslot
- `data-emem-family="<family-key>"` — band family for colour mapping
- `data-emem-responder="<pubkey-b32>"` — responder public key (32B base32-nopad-lowercase)
- `data-emem-verified="true|false|pending"` — Ed25519+BLAKE3 verification state
- `data-emem-href="<endpoint-path>"` — primary endpoint that produced the row

## Trust model

The page implements offline receipt verification in-browser:

```
preimage_bytes = utf8(request_id) | "|" |
                 utf8(served_at)  | "|" |
                 utf8(primitive)  | "|" |
                 utf8(cell0) "," utf8(cell1) "," ... |
                 utf8(cid0) "," utf8(cid1) "," ...
digest         = blake3(preimage_bytes, 32 B)
ok             = ed25519.verify(signature_64B, digest, responder_pubkey_32B)
```

Libraries are pinned esm.sh imports:
- `https://esm.sh/@noble/curves@1.6.0/ed25519`
- `https://esm.sh/@noble/hashes@1.5.0/blake3`

If the libraries fail to load, the page falls back to `POST /v1/verify_receipt`
and labels the receipt row accordingly. A green "verified offline" pill is
only shown when the in-browser path returns true.

## Console pane

Every `fetch()` the page issues prints to a live console at the bottom (or
bottom sheet on mobile). Each row shows:

- ISO timestamp (HH:MM:SS)
- HTTP method
- Path (clickable → opens raw response in a new tab)
- Truncated request body (when applicable)
- Status code, latency in ms
- Truncated response body or error reason
- Per-row pivots: copy as `curl`, copy as Python (`httpx`), copy as MCP `tools/call`, or replay

Press `~` or click the chip in the top-right to focus the console. Press
`Cmd+K` (or `Ctrl+K`) to open the command palette. Press `T` to open the
live try-it drawer. Press `?` for the keyboard shortcut sheet.

## Try-it drawer (`T`)

A slide-out drawer exposes 20 primitives the page knows about (the full
agent surface has 55 MCP tools — discover at `GET /v1/tools`). For each:

- pre-filled JSON request body seeded from the corpus this responder
  actually serves (densest cell, top band)
- per-primitive presets (e.g. "flood risk" / "wildfire smoke" / "climate
  haven" for `ask`)
- one-click Run → live POST to this responder → response, signed receipt,
  in-browser ed25519 verify
- copy-as-curl, copy-as-python (httpx), copy-as-MCP-`tools/call`

This is the outbound counterpart to the reactive console pane: the console
shows calls that already happened, the drawer is where you make new ones.

## Views

- `field` — constellation: every attested cell as a star, positioned by 2-D
  PCA over the `geotessera` 12-byte embedding prefix when available, or by
  Hilbert-projected lat/lng otherwise. Brightness ∝ log(fact_count + 1),
  hue ∝ band family. Drag to lasso a polygon → calls `POST /v1/recall_polygon`.
- `graph` — ontology diagram of the protocol's data model:
  `Source → Materializer → Cell × Band × Tslot → Fact → Receipt`, plus the
  algorithms and manifests anchoring the right edge. Every node shows the
  live count and the primitive that operates on it; click any node to open
  the try-it drawer pre-configured for that primitive.
- `registry` — eight explainable manifest cards (one per manifest:
  registry, bands, algorithms, sources, schema, topics, lcv-1, alphabet).
  Each card carries a plain-English description, count, CID, and a button
  that opens the try-it drawer for the relevant GET endpoint.
- `log` — Sigstore-Rekor-style scroll of recent attestations sourced from
  `/v1/coverage_matrix`. Each row is independently verifiable.

## Glossary

## Agent's-eye view

Every right-pane card (cell recall, ask result, lasso polygon, …) carries
a `human` / `raw` toggle in its top bar. Click `raw` to see the literal
JSON envelope this UI received from the responder — the same payload an
MCP-driven agent would receive at `tools/call`. This makes the page's
claim of being "built for AI agents that humans can also see" literal: a
human can verify the agent surface at any time without leaving the page.

## Foundation encoders

Four GPU encoders co-resident on a 20 GB VRAM budget, surfaced as their
own bands AND woven into algorithm consensus voting:

- **Clay v1.5** — band `clay_v1`, 1024-D CLS, Sentinel-2 L2A 10 bands, ~12 ms warm. Teacher (DINOv2 `vit_large_patch14_reg4_dinov2.lvd142m`) pre-staged at boot.
- **Prithvi-EO-2.0-300M-TL** — band `prithvi_eo2`, 1024-D CLS, HLS V2 6-band, ~13 ms warm.
- **Galileo** — band `galileo`, variant selectable via `EMEM_GALILEO_VARIANT` (default `base`; `tiny` and `nano` also available). The advertised capability is `galileo-<variant>` in `/v1/capabilities.extensions[]`. Sentinel-2 modality wired (multimodal scaffold present; S1 / ERA5 / SRTM / VIIRS / Dynamic-World / WorldCover / LandScan / location zero-masked).
- **JEPA v2 dynamics** — band-synthesizing predictor, currently untrained. Short-circuits to the last attested Tessera vintage; receipt carries `via: short_circuit_untrained` and `untrained_baseline`.

## Triple-consensus algorithms

Three encoders with independent receptive-field aliasing (Clay 2.56 km RF, Prithvi 6.7 km RF, Tessera per-pixel) vote on a change index. Consensus across all three is signal; one-or-none is receptive-field artifact. Surfaced as:

- `clay_prithvi_tessera_triple_consensus@1` — base year-on-year change, gate=0.15 (tunable via `parameters` block)
- `deforestation_triple@1` — adds Hansen GFC mask uplift, `hansen_confirmed` highest-confidence verdict, gate=0.20
- `wetland_change_triple@1` — Tessera leg substituted by JRC GSW recurrence delta, gate=0.10
- `urban_expansion_triple@1` — adds Overture buildings delta + s2.B11 SWIR corroboration, gate=0.20
- `disaster_anomaly_triple@1` — spatial (no temporal recipe); cell-vs-8-neighbour-mean with 2-σ adaptive z-score gate
- `climate_archetype_triple@1` — 12-class Köppen-Geiger classifier; per-encoder centroids seeded from Beck et al. 2018 type localities at `crates/emem-core/data/climate_archetype_centroids_v1.json`
- `coastal_erosion_triple@1` — bathymetry-clamped to active-coastline cells (-5 m ≤ depth ≤ +5 m); gate=0.12

## Algorithm provenance

Every algorithm in the registry carries:

- `formula` — plain math the agent can read and apply
- `inputs` — band keys with role + explanation
- `when_to_use` — agent-targeted trigger guidance
- `citation` — peer-reviewed source
- `accuracy_band` — honest precision estimate
- `parameters` — typed tunable thresholds (gate, k, timeout, ...)
- `learned_from` — citation provenance for every tuned number; an auditor traces any gate threshold back to its referee

Algorithms with an `evaluation: Expr` AST are also re-executable in-process: the responder walks the AST against the snapshot recall and returns a signed composite scalar that any third party with matching `algorithms_cid` + input fact CIDs reproduces deterministically.

## Signed Absence

When a band has no data at a cell, the responder returns a signed Absence fact with a typed reason (`unavailable_capability`, `outside_coverage`, `gpu_unavailable`, `archetype_seed_unavailable`, ...) — not a 404, not an empty array. "We don't have this here" is itself a citable receipt.

## JSON twin

`GET /humans.json` returns the structured snapshot the page renders from: manifests, counts, foundation-encoder inventory, consensus-algorithm summary, endpoint catalog, trust model, and the responder pubkey. Use it as a static fallback when JS cannot run.
