Every recall this protocol serves comes back wrapped in a receipt the responder signs with its Ed25519 key. The receipt commits to content-addressed fact CIDs over a canonical preimage, so the same question asked years apart, at any responder, returns the same bytes — or a verifiable absence. This page walks the trust path live: locate a place, recall a band, verify the signature, and reproduce the CID. Each step issues a real HTTP call against this responder; the JSON you see is what your agent would see.

step 1

locate

POST /v1/locate

Resolve free text into a single cell64 — the protocol's spatial primary key. The locator is a five-layer cascade (wide-bbox table → embedded gazetteer → GeoNames 68 k → sled cache → Photon → Nominatim). It returns the resolved label, lat/lng, and the path it took. 

—
step 2

recall

POST /v1/recall

Pull the named band at that cell. The response carries the value, the band's metadata, the source URLs the responder fetched from, and a signed receipt with fact_cids[], a merkle root, and an Ed25519 signature over the canonical preimage. 

—
step 3

verify

POST /v1/verify_receipt

Recompute the canonical preimage blake3(request_id ‖ served_at ‖ primitive ‖ cells ‖ fact_cids) — sections joined by |, list elements followed by , — and check the Ed25519 signature against the responder's public key. The responder does it here for convenience; the /verify page does the same math client-only with @noble/ed25519 so you don't have to trust this server. 

—
client-only verify → /verify
step 4

reproduce

POST /v1/recall

Call recall a second time. The fact CID is BLAKE3 over canonical CBOR — same cell, same band, same tslot, same derivation key → same 32-character base32 address. If the bytes you receive don't hash to the original CID, something tampered with the response.
