Earth memory protocol · for AI agents

emem

Cite-able, content-addressed, signed memory of every place on Earth. When the user asks a place-question — has this site flooded? what's the elevation? is this farmland? — your agent calls emem_locate then emem_recall, gets a signed fact + receipt, and quotes the receipt. No keys, no signup, no rate limits.

Live data right now: Sentinel-2 (12 spectral bands + 9 indices), Sentinel-1 RTC radar (γ⁰ VV, all-weather), JRC Global Surface Water (1984–2021 flood-recurrence climatology), MET Norway weather (15-min cadence), Tessera 128-D foundation embedding, Copernicus DEM, GMRT topo-bathy, MODIS NDVI, Overture buildings/places/roads. Receipts signed by 777er3yi… at /health; verify offline at POST /v1/verify_receipt.

Worked example — "Has this place flooded?"

# 1. Place → cell64
curl -sX POST https://emem.dev/v1/locate \
  -H 'content-type: application/json' \
  -d '{"q":"23.349668, 85.281371"}' | jq '.cell64, .data_at_this_cell.live_bands_by_topic.flood_history_long_term'
# → "damO.zb000.waro.zcb89"
# → ["surface_water.recurrence"]

# 2. Recall the flood-history band — auto-materializes from JRC GSW.
curl -sX POST https://emem.dev/v1/recall \
  -H 'content-type: application/json' \
  -d '{"cell":"damO.zb000.waro.zcb89","bands":["surface_water.recurrence"]}' \
  | jq '.facts[0].value, .receipt.fact_cids[0]'
# → 0       (this 30 m pixel never sat under water 1984–2021)
# → "zft7…"  ← cite this CID in the agent's reply

# 3. Cross-check with current radar (sees through cloud) and water indices.
curl -sX POST https://emem.dev/v1/recall \
  -H 'content-type: application/json' \
  -d '{"cell":"damO.zb000.waro.zcb89","bands":["sentinel1_raw","indices.ndwi"]}'

Protocol model

Every fact about every place is keyed by (cell, band, tslot) — a 64-bit cell ID over a Hilbert-packed global grid (grid_info), a band name from the content-addressed registry (bands), and a fixed-cadence tslot. Each Fact is one of three kinds: Primary (raw measurement), Derivative (computed from upstream Primaries with the derivation chain attached), or Absence (signed assertion that no data exists at this address, with a reason). Every fact is content-addressed via cid = blake3(canonical_cbor(fact)) — same bytes anywhere on the network resolve to the same CID.

Every read returns a Receipt signed with ed25519 over blake3(request_id ‖ served_at ‖ primitive ‖ cells ‖ fact_cids). The responder pubkey is published at /health and /.well-known/emem.json; any client can verify a receipt offline with the public key alone — no callback to emem.dev. Two clients issuing the same request get the same fact CIDs; receipts are reproducible across operators and time.

The agent loop is three calls: locate (place → cell64) → recall (cell → facts + receipt) → cite receipt.fact_cids[0] in the reply. The protocol is federated: any operator may publish a node, manifests are CID-pinned, and a fact attested under an old bands_cid remains valid forever — supersedes propagate forward, never invalidate the past. No API keys, no signup, no rate-limit-by-credit-card at the request path. Default build is pure Rust — open data fetched via HTTPS Range reads, not Python or GDAL.

Offline receipt verification

# Step 1: any read returns a Receipt.
RESP=$(curl -s -X POST https://emem.dev/v1/recall \
  -H 'content-type: application/json' \
  -d '{"cell":"damO.zb000.xUti.zde78","bands":["copdem30m.elevation_mean"]}')

# Step 2: feed the receipt back to /v1/verify_receipt.
echo "$RESP" | jq '{receipt}' | curl -s -X POST https://emem.dev/v1/verify_receipt \
  -H 'content-type: application/json' --data-binary @-
# → { valid: true, signer_pubkey_b32, preimage_blake3_hex, served_at }
#
# The signer is reproducible from /health (or /.well-known/emem.json):
# 777er3yihgifqmv5hmc2wwmyszgddzderzhsx6rex4yoakwomvka. Anyone with the
# receipt and the pubkey can verify offline using ed25519 — no network call
# back to emem.dev is required.

Address algebra (token cost)

  • cell 64-bit → cell64 4 bigrams · ≤ 4 tokens
  • tslot u64 → t.<base32> · ≤ 2 tokens
  • vec 1792-D fp16 → vec64 12-byte prefix · ≤ 3 tokens
  • cid 32-byte → cid64 8-byte prefix · ≤ 3 tokens

Voxel layout — 1792-D fp16 per cell

Every cell × tslot has a canonical 1792-dimensional fp16 voxel (~3.5 KB / cell). Each band claims a stable byte-offset range so agents can splice multi-band features without de-serialising: geotessera at offset 0..128, a 576-dim foundation slot reserved for forward-compat at 128..704, sentinel2_raw at 704..714, sentinel1_raw at 714..716, dem, landcover, climate, indices, spatial / temporal Fourier, vision encoders, terrain_derived, phenology, nightlights, population, surface_water, koppen, cop_dem, soilgrids, etc. The full layout is content-addressed — not hardcoded — and pinned per Attestation as bands_cid; fetch it once via /v1/bands and cache by CID. The 1792-D voxel is also directly addressable: emem:vec/<vec64> dereferences via /v1/find_similar to the top-k cells whose attested embedding has highest cosine similarity to the addressed vector. Source.scheme = "emem.cube.v1" identifies the canonical layout.

Storage + transport

  • Merkle log: append-only, 1 GiB segments, replay-verifiable.
  • COG reads: hand-rolled TIFF/IFD parser + flate2 (Deflate) + weezl (LZW) + predictors 1/2/3 (the f32 byte-plane shuffle MPC's Sentinel-1 RTC ships) over HTTPS Range — ~300–700 KB per cell from a multi-GB scene, no GDAL.
  • Source list (per band, per request) at /v1/coverage_matrix; satellite/sensor lineage at /v1/fleet.
